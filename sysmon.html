Understanding Sysmon for Threat Hunting

Sysmon or System Monitoring is a Windows System Service that monitors and log system activities to the windows log event. It provide detailed information about process creation, network connections and changes to file creation time. It collects the events and by using SIEM tools, it analyzes them, which further helps in identifying malicious activities in the network.


Important Event IDs:


Event ID 1 - Process Creation
The goal of this lab was to understand how Sysmon Event ID 1 logs process creation events and how parent-child process relationships can be used during investigations.
Environment:
-Windows 11
-Sysmon
-Elastic Agent
-Kibana
-Elastic Search
What is Sysmon Event ID 1?
Sysmon Event ID 1 records every process created on a Windows system. The event provides information such as:
-Process Name
-Executable Path
-Command Line
-Parent Process
-User Account
-Process GUID
Generating Test Events
To generate Event ID 1 logs, Command Prompt was opened and the following command was executed:

  • start chrome
  • whoami
  • ipconfig
  • notepad
  • powershell
  • Get-Process calc
Kibana Query
The following KQL query was used: event.dataset : "windows.sysmon_operational" and winlog.event_id : 1
Why This Matters
Parent-child process relationships are an important investigative artifact.
Examples of normal behavior include:
explorer.exe → chrome.exe
explorer.exe → notepad.exe
Examples of suspicious behavior include:
winword.exe → powershell.exe
outlook.exe → cmd.exe
powershell.exe → rundll32.exe
These relationships can indicate malicious document execution, phishing attacks, or malware activity.
Lessons Learned
-Sysmon Event ID 1 provides detailed process creation telemetry.
-Parent-child process relationships help determine how applications were launched.
-Kibana can be used to quickly identify and investigate process execution activity.
-Understanding normal process relationships is critical before identifying malicious behavior.

Event ID 3 - Network Connection

The objective of this lab was to understand how Sysmon Event ID 3 records network connection activity and how these events can be investigated using the Elastic Stack.
What is Sysmon Event ID 3?
Sysmon Event ID 3 records network connections initiated by processes running on a Windows system.
Unlike traditional firewall logs, Event ID 3 identifies the exact process responsible for the network activity, allowing analysts to associate connections with specific applications.
Event ID 3 can provide information such as:
-Process Name
-Executable Path
-Source IP Address
-Source Port
-Destination IP Address
-Destination Port
-Protocol
-User Account
This information is valuable when investigating malware activity, command and control communications, suspicious PowerShell usage, and unauthorized outbound connections.
Initial Investigation
After successfully investigating Sysmon Event ID 1 (Process Creation), the next step was to generate Event ID 3 network connection events.
Several applications were opened and internet activity was generated. However, no Event ID 3 events appeared in either Event Viewer or Kibana.
At this stage, the Elastic Stack appeared to be functioning correctly because Event ID 1 logs were successfully reaching Kibana.
The issue appeared to originate from Sysmon itself.
Troubleshooting & Remediation
To investigate further, the current Sysmon configuration was reviewed using:
sysmon64.exe -c
The output revealed:
Network connection: disabled
No rules installed
This explained why no Event ID 3 events were being generated.
Although Sysmon was installed and collecting Process Creation events, Network Connection monitoring was disabled.
To enable additional telemetry, the default Sysmon configuration was replaced with the Olaf Hartong Sysmon configuration.
After applying the configuration, Sysmon reported:
Network connection: enabled
This confirmed that Event ID 3 collection had been activated.
Generating Test Events
To generate network connection activity, PowerShell was used to make outbound web requests:
Invoke-WebRequest https://github.com
Additional testing was performed by opening applications such as Discord and Chrome, both of which establish outbound internet connections.
Kibana Query
The following KQL query was used:
process.name : "powershell.exe" and winlog.event_id : 3



Why This Matters
Event ID 3 provides visibility into process-level network activity.
For example:
Github.exe
↓
Connected to
↓
162.159.x.x
or
powershell.exe
↓
Connected to
↓
External IP Address
Without Event ID 3, investigators may know that network traffic occurred, but they may not know which process initiated the connection.
This visibility is especially useful during incident response investigations involving:
-Malware infections
-Command and Control (C2) communications
-PowerShell abuse
-Living-off-the-land techniques
-Unauthorized outbound traffic
Lessons Learned
-Sysmon Event ID 3 must be enabled through configuration before network events are collected.
-Validating the log source is an important troubleshooting step when expected telemetry is missing.
Event ID 3 identifies the specific process responsible for a network connection.
-Understanding normal process relationships is critical before identifying malicious behavior.
